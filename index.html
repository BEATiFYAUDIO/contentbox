Why ContentBox

How it works

We keep the user-facing story simple, but the engine is built for verifiable delivery and frictionless playback. The emphasis stays on outcomes: fast play, clear ownership, and clean unlock.

1 Publish

Local content is packaged into a hash-addressed manifest. Policy and pricing attach to the manifest, and integrity checks allow caching and verifiable delivery.

2 Resolve

Buyers resolve the seller LAN-first when possible. Remote resolution can be configured later. Cached host and manual entry serve as fallbacks.

3 Play

Playback streams with HTTP Range for instant start and seeking. Previews are optional. Entitlement tokens gate access for paid content.

4 Settle

Splits route value to the right people. Payments are sats and zaps oriented. For now, the UI can be stubbed while P2P and playback land first.

Content and attachments

Library experience

The library is the home base. It should feel fast, predictable, and local. When network features exist, they should enhance, not replace, the core experience.

Storefront and access

Buyers should be able to purchase like a file or consume like modern streaming. ContentBox supports both trajectories without forcing a single distribution model today.

Splits and invites

Invite collaborators, collect approvals, and lock split versions before paid exposure. This reduces disputes and keeps stakeholder expectations aligned.

Royalties visibility

Make participations obvious. Owners see what they own. Invitees see what they were granted. Settlement can follow the locked agreement.

Payments, sats and zaps

Payments are designed around sats and zaps. During P2P and playback MVP, a payment stub can keep the UI and user journey intact while the network layer hardens.
