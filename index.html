Mission

ContentBox exists to make platforms optional. We build creator-owned infrastructure so publishing, distribution, and payment can happen without gatekeepers, opaque rule changes, or insider capture. If you have the link, it should work.

ContentBox exists to restore dignity, agency, and trust to digital creation. The modern internet concentrated power in intermediaries. They can shadowban, demonetize, de-rank, de-platform, or rewrite terms without meaningful recourse. Creators become inventory. Audiences become data.

We reject that model. Infrastructure is moral. The way systems are designed determines who holds power, who bears risk, and who benefits when things go wrong.

  • Creators should own their work — local-first by design.
  • Distribution should not require permission — direct links, no gatekeepers.
  • Payments should be honest and hard to manipulate — designed for Bitcoin/Lightning receipts (optional).
  • Systems should be inspectable — standard protocols, visible failure modes.
  • Audits should be possible without coercion — local-first logs, exportable and mergeable.

In one sentence: ContentBox is infrastructure for creators who believe the internet should work for people, not gatekeepers.

ContentBox is creator-owned distribution and monetization infrastructure designed around one loop: play instantly, then unlock optionally. Delivery stays on proven web primitives (HTTP Range). Payments/receipts are isolated as pluggable modules.

  • Lower friction: browser buyers can listen without installing an app.
  • Higher creator control: local-first hosting + explicit entitlements.
  • Resilient architecture: works with one node and scales P2P.
  • Monetization-ready: Bitcoin/Lightning receipts, with clear enforcement hooks.
  • Compliance-friendly posture: no on-chain media storage required, no forced marketplace, no heavy DRM.

How it works

ContentBox publishes a hash-addressed manifest. Buyers open a link. The seller is resolved (LAN first; remote via host/port), and playback streams over HTTP Range for instant start and seeking.

Publish

  • Local content → manifest with hashes.
  • Integrity checks enable caching and verifiable delivery.
  • Price and policy attach to the manifest.

Resolve

  • LAN: mDNS/Bonjour (contentbox._tcp).
  • Remote: DDNS + port forward (optional).
  • Fallback: cached host or manual entry.

Play

  • HTTP Range streaming (instant start).
  • Seek works via 206 Partial Content.
  • Preview + entitlement tokens gate access.

P2P principles

Bytes move peer-to-peer. Control stays with the creator. Optional helpers can improve WAN reliability later. Playback never depends on blockchain.

Discovery

  • LAN-first discovery for zero-config at home.
  • Remote sharing via host/port when needed.
  • Optional rendezvous/relay later for NAT pain.

Trust

  • Integrity by hashes, not platform promises.
  • Identity-aware peers (email now; web3 later).
  • Auditable actions: publish, splits, clearance, settlement.

Access

  • Preview tokens for “play first”.
  • Payment slugs/receipts for unlock later.
  • No heavy DRM; permission + audit.

FAQ

Do buyers need to install ContentBox?

No. Buyers can stream in a browser from a direct link. Installing ContentBox is for creators and power users.

Is media stored on-chain?

No. Playback uses HTTP Range. Blockchain is only for payments/receipts and optional audit anchoring later.

Is this DRM?

No heavy DRM. The goal is permission + integrity + auditability. Enforcement hooks exist, but reality is respected.

What about derivatives and clearance?

Derivatives can be previewed privately by authorized stakeholders, but require OG clearance before public release.
